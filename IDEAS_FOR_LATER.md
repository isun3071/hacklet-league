## Format & Scoring

- **HackLet Agentic (second format).** The second format under the multi-format institution, built when the agentic-coding paradigm warrants it and Classical is operating well. Round shape: 5-min opening + 15-min spec + **48-min supervised agent execution** + 15-min hardening + 15-min eval/prep + 30-min pitches + 30-min deliberation/awards ≈ **~158 min**. Budgets: **500k tokens (CoT included)**, **50k tokens per-prompt cap**, **200 accepted-change cap**, **50 fuzz budget** (spec + hardening only, not during agent execution). The 60-min runtime cap from the earlier sketch is dropped — redundant with wall-clock. Numbers come from real Claude Code data: substantive agent operations burn 30-50k tokens at 5-8 min wall-clock each, so 500k allows roughly 15-20 operations with strategic restraint required. Substrate differs from Classical: a league-built, signed, locked VSCodium extension (chat sidebar + accept/reject UI) instead of the portal chat window — the accept/reject step is the format's deliberate-action friction. Shares the resilience catalog with Classical; the runner is format-agnostic. New categorical award: **Best Direction**. *Why deferred:* big standalone build (Stage 11). *Context:* the agentic paradigm shift threatens a single-format competition but not a multi-format league.

- **Six-axis Resilience Subscore (rename strategy).** Rename "Fuzz" → "Resilience" — "Fuzz" is a technique, "Resilience" is the property being measured. Expand from 2 axes (Security, QA) to 6 by adding Performance Under Load, Race Conditions, Behavioral Consistency, DoS Resistance. Each axis is goodhart-resistant by nature; pool them via weighted normalized composition into a Resilience Subscore that sits next to the Communication Subscore (pitch + cross-ex). Strategic value is real: multi-axis is much harder to game than single-axis as agents improve at self-fuzzing, maps to how senior engineers actually evaluate code, and creates seven new categorical awards (Most Hardened, Most Polished, Most Performant, Most Atomic, Most Deterministic, Most Robust + composite Most Resilient). Ship Season 1 with just Security + QA Fuzz under the "Resilience" branding; add the rest one per season as catalogs mature. *Lands in format_spec.md §4 post-pilot.*

- **Boolean gates, not optimization targets (design principle).** Anywhere a measurement is dominated by threshold psychology — humans perceive cliffs, not slopes — prefer boolean gates over optimization targets. Concrete application: add TTFB ≤ 3s, FCP ≤ 5s, TTI ≤ 10s to QA Fuzz as binary gates. Apps that fail are effectively unusable; apps that clear get small positive contribution with no marginal credit for being faster. Production-grade Web Vitals (LCP < 2.5s as optimization metric) stay out of scope. Goodhart-resistant because players optimize to the gate then redirect time to other axes — no benefit to chasing sub-2-second FCP in a 24-minute build. *The principle generalizes; worth holding in mind for future catalog work.*

- **Surface-pluggable runner architecture.** The runner should think in interaction *surfaces* (protocol, input shape, state effects, auth boundary) rather than HTTP endpoints. Tests apply by capability profile — SQL injection tests fire against any surface reaching a database, whether HTML form, REST API, WebSocket handler, or GraphQL mutation. Discovery modules added over seasons: HTTP forms + REST (S1), GraphQL (S2 via introspection), WebSocket (S2-3, with a league-published introspection convention), SSE (S3), WebRTC signaling (S4), webhooks (S4). *The architectural payoff is durability — today's HTTP focus isn't structurally bound to HTTP. Lands in FUZZ_RUNNER_SPEC.md when the runner is built.*

- **Universal arsenal as catalog foundation.** The resilience catalog's structural foundation is the universal arsenal — tests that hit any deployed web server regardless of language/framework/protocol. Rough count: Universal QA ~50-100 tests, Universal Security ~30-60, Universal Performance ~15-25, Universal DoS ~15-30, Universal Behavioral Consistency ~10-15, Universal Race Conditions ~10-15. **Total ~130-245 language-agnostic tests.** Framework-specific tests *add* to this foundation (Spring patterns, Laravel patterns, ASP.NET patterns — each ~30-100 on top). Implication for Tier 3: C/C++ players get 130-245 applicable tests (the universal foundation), not the 30-40% I was implying earlier. Real coverage is 50-70%. What separates Tier 1 from Tier 3 isn't test count, it's productivity — Tier 1 frameworks handle many universal tests by default; Tier 3 requires manual defense of everything. *The universal arsenal is the league's primary catalog moat — survives framework churn, survives protocol evolution, durable across seasons.*

- **Parameter control philosophy.** Players control model parameters (reasoning depth, temperature, top-p/k, system prompts, tool definitions for Agentic). League controls model selection (one per season), token budget, per-prompt cap, accepted-change cap. Substrate equality means *same model with same available parameters*, not identical parameter usage. Locking parameters would contradict the two-principle thesis ("we don't legislate how players use AI"). The token budget enforces parameter discipline naturally — players who use max reasoning on easy prompts run out before the hard ones. Bonus: parameter choices become legitimate cross-ex material — "You used max reasoning on the auth check, what were you worried about?" *Lands in format_spec.md §5.3 and the eventual Agentic spec.*

- **Hybrid objective + subjective scoring is structurally rare.** Worth naming as positioning argument: HackLet's scoring is unusual. Most formats are pure-objective (track, swimming, chess, FMWC) or pure-subjective (figure skating, MasterChef, debate). Partial-hybrids exist (gymnastics D-score + E-score) but both components are human-judged. HackLet's structure is distinct — resilience is fully machine-measured (no human in the loop); communication is fully human-judged; both axes operate cleanly without contamination. This matches software engineering's actual dual structure: code works or doesn't (objective), engineers explain their work or they don't (subjective). Senior engineering needs both; junior survives on objective alone. *HackLet measures exactly the senior-level cluster. Lands in format_spec.md §4 or §10 when next revised.*

- **Non-technical judge ("Stakeholder Judge") addition.** Add a fourth judge role scoring the pitch axis specifically. Calibration: "2 degrees of separation" — has heard of coding but doesn't code, has heard of AI repeatedly but doesn't fine-tune models, is a professional adult whose work touches tech without being tech. Profile fits: PMs without engineering background, marketing/brand at tech companies, tech journalists, designers who collaborate with engineers, sales engineers, non-engineering founders, lawyers handling tech IP, adjacent-field educators. *Not* "literally any human" (too noisy) and *not* other engineers (too close). Two failure modes this catches: *jargon wall* (engineer talks technically without unpacking, stakeholder locked out) and *infantilization* (engineer overcorrects, patronizes). The skill between them is real-time audience modeling — career-determining at senior levels. Scope: scores pitch quality only, not resilience or cross-ex. *The role self-selects sustainable judge corps via network value + signaling, and adds broadcast-vocabulary delivery moments. Deferred to Season 2+; lands in format_spec.md §3.3 when introduced.*

- **Player accountability principle.** The substrate offers options with different productivity profiles. The player picks; the format scores honestly against the catalog regardless. A fresh grad who picks C++ because "we learned it in class" but can't direct the AI for memory-safe patterns will fail and score low — honest consequence, not unfair handicap. A strong C++ player who picks deliberately, prompts well, reads AI output critically can score meaningfully despite productivity disadvantage. Consistent with the two-principle thesis: substrate equality means *available*, not *productivity parity*. The credential gets information-rich because substrate choice signals specialization. *Lands in format_spec.md §5.1 or §10 — explicit articulation of an implicit principle.*

- **HackLet Systems as future format candidate.** Separate format for systems programming, distinct from Classical's web focus. Substrate: C / C++ / Rust / Zig. Build a small systems program in 24 min (file parser, protocol implementer, memory allocator). Resilience axes around memory safety (ASan, Valgrind, AFL fuzz mutation), performance, correctness, crash resistance. Separate format rather than Classical extension because web resilience and memory safety are different skill clusters that shouldn't share a composite. *Build after Classical is operational and Agentic is launched — probably 5+ years out. Preserves C/C++ as competitive substrate without diluting Classical's web focus.*

## Substrate & Catalog

- **Majority intersection substrate curation (with final Season 1 selection).** Two-clause filter: (1) language appears in ≥ N% of fresh-grad SWE *aspirational* resumes (the resumes CS undergrads write to break into entry-level web SWE — not all SWE resumes, which would dilute with COBOL maintainers); (2) league maintains fuzz coverage across all six axes for the dominant framework(s). Both required for Tier 1. The "majority intersection" framing beats union (union grows monotonically toward the universe) and beats strict intersection (approximately empty). The critical convergence property: in this domain, audience interest and testability *align* — the same forces that put a language on aspiring engineers' resumes make it testable (most training data, most security research, most framework standardization, most documentation). HackLet doesn't have to choose between "what players want" and "what we can test." Threshold for S1: N ≥ 50%.

  **Final Season 1 substrate:**

  *Tier 1 (full coverage):* Python (Flask, FastAPI, Django, Starlette), JavaScript (Express, Fastify, Hono, Koa), TypeScript (same Node ecosystem), Java (Spring Boot, Quarkus, Micronaut), Go (net/http, Gin, Echo, Chi, Fiber), PHP (Laravel, Symfony), C# (ASP.NET Core with Identity + EF Core).

  *Universal substrate:* HTML, CSS, vanilla JS, SQL (PostgreSQL, SQLite, Redis Tier 2).

  *Tier 3 (universal-only testing, diversity signaling):* C (via libmicrohttpd, mongoose), C++ (via Crow, Pistache). The universal arsenal applies because any deployed HTTP server presents interactable interfaces. Roughly 50-70% catalog coverage — meaningfully participatory, not throwaway.

  *Excluded:* Rust (below threshold for junior web roles, framework ecosystem still maturing), Ruby (declining), Swift/Kotlin (mobile-focused), niche languages.

  *Lands in format_spec.md §5.4 post-pilot. Two-clause filter generalizes to frameworks, databases, auth patterns — same principle at every substrate layer.*

- **Methodology transparency for substrate decisions.** Publish per season: data sources sampled (LinkedIn Easy Apply entry-level SWE filters, HackerRank student survey, Stack Overflow student responses, FAANG/tier-2 new grad listings, university career services data, hackathon repos), sample sizes, threshold N%, resulting Tier 1/2/3 placements. Makes curation auditable. Defends against "favoritism" criticism — "Python is N% of target resumes, well above our Y% threshold" beats "we like Python." Matches academic transparency norms.

## Broadcast & Production

- **Process-as-spectacle broadcast theory (MasterChef pattern).** The build phase IS the show, not just a precursor to pitch+crossex. AI substrate compresses the build to televisable duration. Screen capture is richer than cooking-show overhead cameras (audiences directly observe creative work). Real-time metrics provide commentary substance — without them, commentary is surface-level "she's typing fast" filler. The resilience score is the commentary backbone. From MasterChef specifically: three-criteria scoring per category (substance/coherence/delivery for pitch; defense substance/code grounding/composure for cross-ex), quantitative scores within subjective evaluation, multi-judge specialization with disagreement as content, deliberation as its own broadcast phase. What we don't borrow: reality-TV theatrics, celebrity-judge personality center, elimination dynamics. *Lands in format_spec.md §3.9 when expanded; LEAGUE_OPERATIONS for the two-precedent positioning.*

- **Chat log as the killer broadcast asset.** The chat log between player and AI is HackLet's most powerful broadcast asset. Structurally novel — no previous televised creative competition has had access to the contestant's cognition stream in real time. Chess broadcasts show engine evaluation, not player thinking. Cooking shows infer chef intent. HackLet broadcasts the cognition itself. Conversational format is legible to non-technical audiences. Prompts reveal strategy without commentator explanation needed ("scratch that, just test sqli" explains itself). The player-AI dynamic creates relatable interpersonal-feeling drama. Director pattern: 12 screen captures broadcast to production; director picks the most interesting screen at each moment based on prompt activity, fuzz events, visual progress, crisis moments, late-round panic. *Picture-in-picture for parallel comparisons. Commentator dashboard surfaces story candidates. Lands in format_spec.md §3.9 when expanded.*

- **Venue vs broadcast audio separation.** Venue stays quiet (players need to concentrate). Commentary lives in a separate booth, mixed for broadcast only. Matches sports broadcast conventions (tennis, golf, chess). Operational implication for Tier A chapters: commentary booth must be sound-isolated, watching the broadcast feed not the room directly. *Lands in chapter broadcast setup docs when broadcast infrastructure is built (Stage 6).*

- **Vocabulary bootstrapping + cultural-infrastructure framing.** Each broadcast event teaches the audience more engineering spectator vocabulary. By Season 3-5, "defended SQL clean," "burning tokens fighting the model," "scratch the search endpoint" become familiar broadcast vocabulary — the way "deglaze," "fond," "ribbon stage" became familiar through MasterChef. MasterChef built cooking spectator vocabulary over 15 years; HackLet has the same opportunity for engineering, accelerated by the non-technical judge addition acting as on-broadcast vocabulary delivery. *This is the long-term cultural-infrastructure play — bigger than "another competitive format." Engineering hasn't had visible practice rituals before. HackLet positions to change that.*

## Pitch Positioning

- **Four load-bearing claims.** Worth knowing by heart for pitches: (1) AI is the enabling substrate, not a complication — pre-AI, 24 minutes produces toy software; with AI, 24 minutes resembles a real product. The format only exists because AI exists. (2) Hackathons degrade because of AI; HackLet inverts the trajectory — hackathon credentialing weakens as AI capability grows (human contribution becomes ambiguous), HackLet credentialing strengthens (measured skills isolate to direction / verification / defense). (3) The 24-minute format makes engineering finally televisable — MasterChef proves process-as-spectacle, AI compression makes builds fit watchable durations, chat log is the cognition feed. (4) Convergence + divergence makes the format both watchable and comparable — curated substrate enforces enough convergence for apples-to-apples comparison, AI direction creates enough divergence for content.

- **The elevator pitch: "FMWC + MasterChef for vibecoding."** Two precedents handle two halves. FMWC contributes the credentialing skeleton — founded 2020 by Andrew Grigolyunovich post-ModelOff, grew from 125 to 1000+ competitors over 5 years, $10k → $120k prize pool, ESPN3 finals 2021, ESPN8 All-Star Battle 2022, HyperX Esports Arena hosting finals by 2023, real recruiting outcomes. Empirical proof that "objective-skill competitive format with broadcast and persistent rankings" works at niche-credentialing scale. MasterChef contributes the broadcast-of-process pattern — 15+ years on FOX, time-pressured creative work as watchable spectacle, multi-judge structured evaluation, vocabulary bootstrapping over seasons. HackLet's contribution: applying both patterns to AI-assisted defensive coding, made newly possible by AI compressing the build clock from 24 hours to 24 minutes. *"The fuzz is what separates hacklets from slop" is HackLet's irreducible-test equivalent of MasterChef's taste — charisma in pitch doesn't compensate for slop in fuzz, just as presentation doesn't save a bland dish.*

- **Realistic audience trajectory (FMWC-anchored).** Year 1 (UPE pilot): under 500 viewers per event. Years 2-3 (chapter expansion): 1k-3k per event. Years 4-5 (regional/championship maturity): 10k-30k per championship. Years 6-10: ESPN8-tier broadcast deals, $100k+ prize pools, real recruiting pipeline. Year 10+: MasterChef-scale audiences as aspirational upside if vocabulary bootstrapping pays off. Anti-Icarus framing: *we're not flying too close to the sun expecting universal acclaim Day 1; we're building infrastructure and letting the audience find it.* Core viewership segments: aspiring/competing engineers, AI-coding-adjacent tech workers, CS students specifically, recruiters/hiring managers (the keystone audience that makes the credential matter), engineering-curious civilians (the long bet). *Honest framing for chapter operators and sponsors: HackLet's value is audience quality, not quantity. Niche-but-real is a viable position for credentialing institutions, and the FMWC record validates the trajectory.*

- **Tagline candidates.** Three that have tested resonant: "the fuzz is what separates hacklets from slop" (closer, on the site already), "the models do the typing. the judgment is yours." (positioning, on the site already), "AI didn't break the hackathon. AI made HackLet possible." (origin framing, not yet on the site). Reserved here for public-facing copy.

- **CS-student substrate alignment.** Classical's chat-window-and-copy-paste substrate matches the economically-dominant workflow CS students already use. Agentic IDE tooling costs money: Cursor's $20/mo even with student-verified free year requiring SheerID friction, GitHub Copilot Pro paused for new sign-ups since April 2026, Claude Code paid, Cline requires bring-your-own-key. The chat-window workflow is the only fully-free option, so ~70-85% of CS undergrads primarily use it. Classical meets students where they work. *Roughly 3-4 academic years of clean Classical credentialing before the incoming cohort skews agentic-experienced enough to make it feel generationally dated. Already partially landed in format_spec.md §10.*

## Operations & Community

- **FMWC positioning.** The Financial Modeling World Cup (founded 2020 by Andrew Grigolyunovich, Latvia, after ModelOff was discontinued) is HackLet's structural precedent: a niche measurable skill → recurring tiered competition + persistent rankings + an ESPN2 broadcast (All-Star Battle, 2022) → a real institution. Deploy in pitches to chapter operators / sponsors / employers to answer "*is this real?*" — HackLet applies a pattern that already worked, to a larger domain (AI coding > financial modeling in participant pool and cultural pull). Frame as **template, not parity** — HackLet hasn't earned FMWC's reach. *Verify the specific facts before any public use.*

- **Competitive landscape — Microsoft "Agents League" (AI Skills Fest, June 4–14, 2026).** ~10-day, esports-framed hackathon, ~$55k prize pool, combining live AI coding battles, async project submissions, and a Discord community across three tracks (Creative Apps with GitHub Copilot, Reasoning Agents with Microsoft Foundry, Enterprise Agents for Microsoft 365 Copilot). Structurally different from HackLet: a one-off marketing/skills-fest event vs a recurring credentialing institution; criteria-based/subjective judging vs an objective adversarial fuzz catalog; multi-day multi-track vs a 24-min solo compressed round. Honest read: a hyperscaler entering **validates the category** but is also a **distribution threat HackLet can't match** (Copilot's install base). HackLet's defense is not reach — it's the credential + the fuzz moat. Practical effect: **drop "first / pioneering" language** from positioning.

- **Testing-center framing.** HackLet workstations parallel certification testing-center rigor (CCIE lab, CISSP proctoring): controlled substrate, identical conditions, audited integrity. Useful for making the credentialing claim legible — the conditions are *why* the score means something.

## Infrastructure & Security

- **Fuzz-runner sandbox hardening doc.** Before Stage 5, update [FUZZ_RUNNER_SPEC.md](FUZZ_RUNNER_SPEC.md) to explicitly treat the runner as a sandbox executing untrusted contestant code, not just a test executor. Must address: container escape, resource exhaustion, network-egress prevention from submissions, and the hidden-pool-must-never-reach-workstations boundary. *Why deferred:* runner is Stage 5; flagged now so it isn't forgotten. *Context:* identified during the foundational design review.

- **Broadcast architecture (deferred details).** Capture via **VNC pull** from each workstation (league pulls the display rather than the workstation pushing). League stats overlay built as a **Next.js dashboard consumable by OBS as a browser source**. Production complexity scales with event tier (chapter → regional → championship). Specific patterns to encode when this lands: director-selected screen cut (12 players, picture-in-picture for parallel archetypes), commentator dashboard with all-player metrics, venue/broadcast audio separation. *Belongs to Stage 6.*

- **Threat model + security posture (eventual SECURITY.md).** HackLet acknowledges submissions are untrusted code generated by players directing AI. Structural mitigations: workstation network isolation (firewall allowlist) prevents exfiltration and lateral movement; per-player Unix accounts (ephemeral, no sudo) prevent tampering; fuzz runner executes submissions in resource-limited isolated containers; judge clickaround renders in sandboxed browser contexts (CSP-restricted iframes); AI proxy logs every prompt; conduct rules (§8) prohibit malicious intent with disqualification + ban consequences. The threat model is *intentionally limited* to what's structurally realistic — defenses against elaborate jailbreak attempts are deliberately not built, because the competitive context (24-minute clock, audit logs, judge inspection) is naturally adversarial to malicious code authorship. Matches MLH's empirical track record (15+ years without a malware crisis, with less structural isolation than HackLet). *Lands as SECURITY.md when Stage 4-5 implementation begins.*
