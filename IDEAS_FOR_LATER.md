## Format & Scoring

- **Agent interface for unified league substrate (Stage 12 work, post-Sprint launch).** Where the league hosts AI substrate (Tier A and Tier B), the substrate provides *both* a chat-window interface and a league-built signed VSCodium extension with an in-IDE agent (chat sidebar plus accept/reject UI for agent-proposed file changes), modeled on Cline/Roo Code patterns. Both interfaces talk to the same proxy, the same season-pinned model, and the same per-player token budget. Players use whatever combination fits their workflow. The agent interface is *substrate expansion*, not a new format variant — there is no "HackLet Agentic" because there is no Classical/Agentic distinction (see "Two-axis format naming taxonomy" entry). The extension is a substantial standalone build (TypeScript, ~4-6+ weeks for v1) which is why it ships after the chat-only substrate operates well. Adds an **agent freeze rule** that applies whenever the agent is in use: at freeze the workspace reverts to the *last accepted edit*; pending agent proposals are dropped along with partial edits. Consistent with the chat-window freeze behavior (truncated AI responses roll back to pre-prompt state). The accept/reject discipline becomes part of strategic AI direction across the unified substrate (when to delegate to the agent, when to drive with chat-style thinking). Future model rotations may justify additional substrate features (multi-agent coordination, MCP support, structured tool definitions); all are substrate expansions, not new format relationships. *Built only when chat-only substrate is operating well at Tier A/B. Builds on the OpenAI-compatible chat completions endpoint from format_spec §5.3, so the extension is a client of the same substrate the chat window already uses.*

- **Six-axis Slop Score (multi-axis expansion).** The Slop Score is the composite of multiple slop dimensions. Expand from 2 axes (Security, QA) to 6 by adding Performance Under Load, Race Conditions, Behavioral Consistency, DoS Resistance. Each axis is goodhart-resistant by nature; pool their per-axis slop (each deduction-only, lower-is-better) into the composite Slop Score that sits next to the Communication Score (pitch + cross-ex). Strategic value is real: multi-axis is much harder to game than single-axis as agents improve at self-fuzzing, maps to how senior engineers actually evaluate code, and supports seven categorical awards (Most Hardened, Most Polished, Most Performant, Most Atomic, Most Deterministic, Most Robust + composite Most Resilient — award titles stay aspirational even though the score they reward is deduction-only). Ship Season 1 with just Security + QA under the Slop Score; add the rest one per season as catalogs mature. *Lands in format_spec.md §4 post-pilot.*

- **Boolean gates, not optimization targets (design principle).** Anywhere a measurement is dominated by threshold psychology — humans perceive cliffs, not slopes — prefer boolean gates over optimization targets. Concrete application: add TTFB ≤ 3s, FCP ≤ 5s, TTI ≤ 10s to QA Fuzz as binary gates. Apps that fail are effectively unusable; apps that clear get small positive contribution with no marginal credit for being faster. Production-grade Web Vitals (LCP < 2.5s as optimization metric) stay out of scope. Goodhart-resistant because players optimize to the gate then redirect time to other axes — no benefit to chasing sub-2-second FCP in a 24-minute build. *The principle generalizes; worth holding in mind for future catalog work.*

- **Surface-pluggable runner architecture.** The runner should think in interaction *surfaces* (protocol, input shape, state effects, auth boundary) rather than HTTP endpoints. Tests apply by capability profile — SQL injection tests fire against any surface reaching a database, whether HTML form, REST API, WebSocket handler, or GraphQL mutation. Discovery modules added over seasons: HTTP forms + REST (S1), GraphQL (S2 via introspection), WebSocket (S2-3, with a league-published introspection convention), SSE (S3), WebRTC signaling (S4), webhooks (S4). *The architectural payoff is durability — today's HTTP focus isn't structurally bound to HTTP. Lands in FUZZ_RUNNER_SPEC.md when the runner is built.*

- **Universal arsenal as catalog foundation.** The fuzz catalog's structural foundation is the universal arsenal — tests that hit any deployed web server regardless of language/framework/protocol. Rough count: Universal QA ~50-100 tests, Universal Security ~30-60, Universal Performance ~15-25, Universal DoS ~15-30, Universal Behavioral Consistency ~10-15, Universal Race Conditions ~10-15. **Total ~130-245 language-agnostic tests.** Framework-specific tests *add* to this foundation (Spring patterns, Laravel patterns, ASP.NET patterns — each ~30-100 on top). Implication for Tier 3: C/C++ players get 130-245 applicable tests (the universal foundation), not the 30-40% I was implying earlier. Real coverage is 50-70%. What separates Tier 1 from Tier 3 isn't test count, it's productivity — Tier 1 frameworks handle many universal tests by default; Tier 3 requires manual defense of everything. *The universal arsenal is the league's primary catalog moat — survives framework churn, survives protocol evolution, durable across seasons.*

- **Parameter control philosophy.** Players control model parameters (reasoning depth, temperature, top-p/k, system prompts, and tool definitions when the agent interface is in use). League controls model selection (one per season), token budget, per-prompt cap, accepted-change cap. Substrate equality means *same model with same available parameters*, not identical parameter usage. Locking parameters would contradict the two-principle thesis ("we don't legislate how players use AI"). The token budget enforces parameter discipline naturally — players who use max reasoning on easy prompts run out before the hard ones. Bonus: parameter choices become legitimate cross-ex material — "You used max reasoning on the auth check, what were you worried about?" *Lands in format_spec.md §5.3 (substrate); agent-interface tooling is BUILD_ROADMAP Stage 12.*

- **Hybrid objective + subjective scoring is structurally rare.** Worth naming as positioning argument: HackLet's scoring is unusual. Most formats are pure-objective (track, swimming, chess, FMWC) or pure-subjective (figure skating, MasterChef, debate). Partial-hybrids exist (gymnastics D-score + E-score) but both components are human-judged. HackLet's structure is distinct — resilience is fully machine-measured (no human in the loop); communication is fully human-judged; both axes operate cleanly without contamination. This matches software engineering's actual dual structure: code works or doesn't (objective), engineers explain their work or they don't (subjective). Senior engineering needs both; junior survives on objective alone. *HackLet measures exactly the senior-level cluster. Lands in format_spec.md §4 or §10 when next revised.*

- **Non-technical judge ("Stakeholder Judge") addition.** Add a fourth judge role scoring the pitch axis specifically. Calibration: "2 degrees of separation" — has heard of coding but doesn't code, has heard of AI repeatedly but doesn't fine-tune models, is a professional adult whose work touches tech without being tech. Profile fits: PMs without engineering background, marketing/brand at tech companies, tech journalists, designers who collaborate with engineers, sales engineers, non-engineering founders, lawyers handling tech IP, adjacent-field educators. *Not* "literally any human" (too noisy) and *not* other engineers (too close). Two failure modes this catches: *jargon wall* (engineer talks technically without unpacking, stakeholder locked out) and *infantilization* (engineer overcorrects, patronizes). The skill between them is real-time audience modeling — career-determining at senior levels. Scope: scores pitch quality only, not resilience or cross-ex. *The role self-selects sustainable judge corps via network value + signaling, and adds broadcast-vocabulary delivery moments. Deferred to Season 2+; lands in format_spec.md §3.3 when introduced.*

- **Player accountability principle.** The substrate offers options with different productivity profiles. The player picks; the format scores honestly against the catalog regardless. A fresh grad who picks C++ because "we learned it in class" but can't direct the AI for memory-safe patterns will fail and score low — honest consequence, not unfair handicap. A strong C++ player who picks deliberately, prompts well, reads AI output critically can score meaningfully despite productivity disadvantage. Consistent with the two-principle thesis: substrate equality means *available*, not *productivity parity*. The credential gets information-rich because substrate choice signals specialization. *Lands in format_spec.md §5.1 or §10 — explicit articulation of an implicit principle.*

- **HackLet Systems as future format candidate.** Separate format for systems programming, distinct from Vibe's web focus. Substrate: C / C++ / Rust / Zig. Build a small systems program in 24 min (file parser, protocol implementer, memory allocator). Resilience axes around memory safety (ASan, Valgrind, AFL fuzz mutation), performance, correctness, crash resistance. Separate format rather than a Vibe extension because web resilience and memory safety are different skill clusters that shouldn't share a composite. *Build after Vibe is operational and the agent interface has shipped — probably 5+ years out. Preserves C/C++ as competitive substrate without diluting Vibe's web focus.*

- **HackLet Unslop (canonical second format).** The companion to HackLet Vibe under the multi-format institution. Same workstation/timing infrastructure as Vibe, but the starting state differs: the league's substrate generates a deliberately-broken application live during the 5-minute round opening (zero leakage risk by design — the slop literally doesn't exist before the round). All players receive identical slop at T+5:00. **No pre-disclosure of vulnerabilities** — players must find, diagnose, triage, fix, and verify themselves using AI assistance. Commentators *can* discuss vulnerabilities during the round because broadcast audio is separated from the venue (the dramatic-irony broadcast device — audience knows what's broken, watches players find it). Expanded scope covers the full surface of brokenness the fuzz catalog tests: security oversights, QA failures (crashes, unhandled exceptions, edge cases), performance disasters (polytime queries, slow TTFB/FCP), reliability gaps (race conditions, missing transactions), hallucinated package versions or function calls (latent runtime failures), ugly UX. The slop's only requirement is that it compiles, runs, and deploys. Players are also permitted to add their own features, UX overhauls, and aesthetic choices — "make it yours" is part of the credential. Token-cost asymmetry creates strategic depth: diagnosing burns input-heavy tokens (loading codebase for analysis), fixing burns output-heavy tokens (writing corrections). Same fuzz catalog scores both Vibe and Unslop submissions identically at freeze. *Credentials a different skill cluster than Vibe — reading unfamiliar code under pressure, triaging across multiple dimensions of brokenness, AI-assisted vulnerability hunting and verification, integrated engineering judgment. Maps more completely to FDE work patterns. Introduced Season 2-3 after Vibe operates successfully; deserves its own format_spec section then.*

- **Two-axis format naming taxonomy.** HackLet formats are specified by **{Format} {Timer}**. Format axis = Vibe (build from scratch) or Unslop (remediate broken app). Timer axis = XP (12 min, 50k tokens) / Sprint (24 min, 100k) / Scrum (36 min, 150k) / Agile (48 min, 200k) / Waterfall (72-96 min, 300-500k). Token budgets scale with timer. **2 formats × 5 timers = 10 sanctioned variants.** Foundational format: **HackLet Vibe Sprint** (pilot, BU chapter). The earlier three-axis design with a Relationship axis (Classical vs Agentic) was retired in favor of the unified-substrate model — where the league hosts AI, both chat-window and in-IDE agent interfaces are provided with a unified token budget, and players use whatever combination fits their workflow (see "Agent interface for unified league substrate" entry and format_spec.md §1, §5.3). The Relationship distinction was infrastructure-driven, not skill-driven, and removing it makes the taxonomy more honest to the format's "we don't legislate how you use AI" thesis (§10). The compositional structure lets the league add formats without renaming what came before. *Landed in format_spec.md §1. Documented here as the canonical naming framework all future format introductions reference.*

- **Agent freeze rule (unified substrate).** When token budget runs out or time's up, any pending agent proposals are dropped and the workspace reverts to the *last accepted edit*. Chat-window outputs at freeze follow the existing rule (truncated AI responses roll back to pre-prompt state). Both apply at code freeze regardless of which interface the player was using. Player responsibility for budget management, pacing, and accept/reject discipline. Strategic implication: players who batch-accept agent proposals near the freeze risk losing entire chains if the freeze catches them mid-batch. Players who accept incrementally lose less but cap their per-segment scope. The accept/reject discipline becomes part of strategic AI direction across the unified substrate. *Lands in eventual substrate spec when the agent interface ships (Stage 12 work).*

- **No-coddling philosophy (load-bearing strategic positioning).** Format mirrors production engineering constraints honestly. Token exhaustion has consequences. Time management has consequences. Deployment failures have consequences. The agent freeze drops pending changes. Companies want immediately productive juniors with no onboarding runway; HackLet credentials engineers who already function under realistic constraints. The format does not soften failures to make players feel better — it scores honestly so the credential carries market weight. Applied consistently across: Vibe freeze, Unslop freeze, agent freeze, token exhaustion, time management, deployment failures, slop score consequences. *Generalizes as design principle. Worth holding load-bearing.*

- **Categorical awards as multi-track credentialing.** The categorical awards (Most Resilient, Best Communicator, People's Hacklet per round; Best UX/UI, Most Novel, Most Efficient, Iron Player, Comeback Player at tournament level — see format_spec §4) plus Best Overall create multiple distinct credentialing tracks. Each maps to different specialty hiring needs: Most Resilient → AI security engineering; Best Communicator → product management / technical communication-heavy roles; Best UX/UI → design engineering; Most Novel → innovation / R&D; Most Efficient → resource-constrained engineering (post-tokenmaxxing crisis); People's Hacklet → developer relations / community; Best Overall → integrated FDE pipeline. Different sponsors target different tracks based on hiring needs. The credential ecosystem expands without format proliferation. *Implicit today; worth being explicit in sponsor-facing materials when chapters approach sponsors.*

- **Repeat champions as sustained capability signal.** Mastery levels paralleling chess titles. Future structure: HackLet Master (3+ wins across any format), HackLet Grandmaster (5+ wins, with at least one at regional or higher). Sustained performance is more predictive of professional capability than single-event peak. Titles persist across seasons and accumulate across the Vibe and Unslop format families. *Lands once enough cumulative event history exists to make titles meaningful — probably Year 2-3.*

- **Variant pruning — sanctioned operational matrix.** The format taxonomy is **2 formats × 5 timers = 10 sanctioned variants** (Vibe and Unslop at each of XP, Sprint, Scrum, Agile, Waterfall). All 10 combinations are sanctioned because the unified-substrate model (chat window plus agent interface, shared token budget, players choose freely) means there is no infrastructure-imposed reason to disallow short timers with agent-style work. Players at XP timer (12 min) who delegate to the agent will discover for themselves whether the wall-clock fits their substantive agent operations; the format does not pre-decide their tool strategy for them. The earlier "Agentic XP/Sprint/Scrum unsanctioned" carveout disappears with the Relationship axis. The pruning principle still generalizes: the league sanctions only variants where credentialing claims are honest. Future timer values may be added as substrate evolution warrants (e.g., new "Endurance" timer at 4+ hours if a market case emerges). The sanctioned matrix may be revisited if operational data shows certain variants consistently fail to produce meaningful competition. *Landed in format_spec.md §1 as the sanctioned variant count when next revised.*

- **Multi-day Tier A tournament template.** Tier A regional and championship events use individual-competition tournament structure derived from FIDE Swiss-system, Olympic qualification, and golf-cut precedent — *cumulative-score advancement across multi-day qualifying, not bracket pairings* (the format is solo-against-catalog, not head-to-head). The full template:

  *Tournament size and qualifying capacity.* For a 12-round qualifying structure across days 1-2 (2 concurrent pods × 3 time blocks × 2 days), each round runs 8 players. Maximum theoretical capacity is 12 × 8 = 96 player-rounds. Players can compete in maximum 6 of these (one per time block; can't be in concurrent rounds). Minimum participation threshold for qualifying eligibility: **N/4 with a floor of 2**, where N is total qualifying rounds. For 12 rounds, threshold = 3. Maximum starting field at minimum participation: 96 ÷ 3 = **32 players**. Realistic operational capacity is **20-32 players** because average participation tends toward 4-5 rounds per player rather than minimum 3.

  *Day 1-2 schedule template (per day)*: 9:00-9:25 opening/recap ceremony · 9:30-12:00 qualifier rounds 1+2 concurrent · 12:00-13:00 lunch (includes Zamboni reset) · 13:00-15:30 qualifier rounds 3+4 concurrent · 15:30-18:00 qualifier rounds 5+6 concurrent · 18:00+ networking/dinner/decompression. End of day 2 ~19:00: qualifier announcement and Day 3 pod assignment notifications.

  *Scoring across qualifying.* Player's qualifying score = simple average across their participated rounds. Average (not median, not trimmed mean) because (a) simple to broadcast, (b) rewards consistency which matches the credentialing intent, (c) the participation threshold already protects against single-disaster-round catastrophe, (d) median creates strategic-disaster gaming surface. Top 16 (top half of 32-player field) advance to Day 3. Ranks 17-24 designated as **alternates** (next 8 players below qualifying cutoff, capability-matched, available to substitute for no-shows).

  *Day 3 structure: snake-draft semifinals + finals.* 16 advancing players split into 2 pods of 8 via **snake draft on qualifying rank** (pod A picks #1, pod B picks #2, pod B picks #3, pod A picks #4, etc. — produces exact total-seeding-strength balance, e.g. pod A = sum 68, pod B = sum 68 for 16 players). Snake draft is universal (not gradient by tournament tier) because the player UX cost is absorbed by the platform — players see their pod/workstation assignment in their portal without needing to understand the algorithm. Concurrent semifinal rounds 9:30-12:00. Top 4 from each pod (within-pod ranking only, **handles judge calibration variance structurally** — different judge panels per pod don't matter when advancement is within-pod) advance to finals. Finals 13:30-16:00 with **single judge panel scoring all 8 finalists** (preserves cross-finalist fairness). Award ceremony 16:30-17:00.

  *Two-leaderboard model.* The tournament maintains two parallel leaderboards: (1) **Qualifier leaderboard** (32-player) accumulates scores across all rounds each player participated in (qualifying + semifinals + finals for advancers), averaged. Determines tournament-level categorical awards. (2) **Finals leaderboard** (8-player) scores the finals round only. Determines finals-specific awards. The two can produce different Best Overall winners — a consistent-across-the-tournament performer might win Qualifier Best Overall while a finals-peak performer wins Finals Best Overall. Both are real credentials with distinct signals (sustained capability vs peak under championship pressure).

  *Award allocation across leaderboards (Tier A multi-day tournament).* **Qualifier-leaderboard awards (cumulative):** Tournament Best Overall · Tournament Most Resilient · Tournament Best Communicator · Tournament People's Hacklet · Best UX/UI (judge-aggregated across rounds) · Most Novel (judge-aggregated across rounds) · Most Efficient (Tier A token-discipline, cumulative) · Iron Player (maximum 6 rounds participated) · Comeback Player (most improved early-to-late qualifying). **Finals-leaderboard awards:** Finals Best Overall · Finals Most Resilient · Finals Best Communicator · Finals People's Hacklet. ~13 distinct credential-bearing positions in a tournament, supporting specialization recognition without award sprawl because the field is 30-60 players across 3 days.

  *Tag credentialing structure.* Player profiles accumulate tournament credentials at multiple levels: **Champion** (Finals Best Overall winner, 1 player) · **Finalist** (made finals, 8 players) · **Day 3 Qualifier** (made Day 3 but eliminated in semis, 8 additional players for 16 total) · **Tournament Participant** (competed in qualifying, didn't advance). Plus Categorical Award tags stack independently. The information-richness lets employers eventually interpret credentials specifically — Champion + Finals Most Resilient is a different signal than Tournament Qualifier + Most Novel.

  *Alternates policy.* Top 24 of starting field designated as either Qualifier (top 16) or Alternate (ranks 17-24). Alternates report to venue 8:30 AM Day 3, designated specific qualifier they'd replace (rank 17 → highest-rank no-show, rank 18 → second no-show, etc.). Activation cutoff 9:15 AM. Substitution in place without recomputing snake (minor pod-balance impact is structurally acceptable for late-stage changes). Non-activated alternates released with audience recognition and "Tournament Alternate" credential tag. Compensation: free venue access, food + beverages, "Tournament Alternate" tag.

  *No-show policy.* Four categories with proportionate consequences: **Excused absences** (documented emergency, illness, transportation crisis, etc.) — no penalty, retains "Qualified, withdrew due to documented emergency" tag, eligible for future events. Documentation within 7 days post-event, chapter admin good-faith review, superadmin appeal path. **Communicated withdrawal** (notified by 8:00 AM Day 3, doesn't meet excused criteria) — small persistent rating penalty, "Qualified, withdrew" tag, eligible for future events. **No-call no-show** (no appearance, no communication) — substantial rating penalty, "Qualified but disqualified" public tag, one-event chapter cooldown (lifts after one event elapses). Repeat no-call no-shows: extended cooldown, chapter membership review. **Late arrival**: 10-minute grace from round start with reduced build time; after grace, treated as no-show for that round but eligible for later rounds. Public visibility tiered — excused absences private, communicated withdrawals private with small mark, no-call no-shows publicly visible on tournament record, repeat patterns publicly aggregated.

  *Platform absorbs operational complexity.* The platform handles snake-draft pod composition, workstation assignment within pods (randomized for fairness), judge panel composition by available corps, substitution logic, schedule notifications, and credential tag assignment. Players see "Report to Pod A, Workstation 3, 9:30 AM" in their portal — they don't need to understand the algorithms behind it. Chapter operators configure tournament parameters (rounds, pods, judge corps, variant); the platform produces the operational artifacts. This is recurring HackLet design philosophy: *complexity the platform absorbs is complexity that doesn't burden participants*.

  *Judge corps planning.* Multi-day tournament demands a corps of 6-8 senior judges with rotation across pods to manage cognitive fatigue. Concurrent semifinal pods need separate panels (4 judges each). Finals uses a single panel of 4, ideally fresh from the morning or composed from rested rotators. Tester judges, UX designer judges, general engineering judges, and stakeholder judges drawn from the chapter's recruited corps per the format_spec §3.3 specifications.

  *Scope and timing for HackLet.* This is **Tier A territory only** — Tier C/B events stay single-round or single-day. The platform infrastructure to support the template lives in BUILD_ROADMAP Stage 8 (Federation Features). Realistic deployment is Year 2-3 of league operations, after first Tier A chapters complete verification and accumulate single-day Tier A event experience.

- **Format-lane structure (Option C — format-specific tournaments + cross-format championships).** Multi-format institution coordination matches FIDE chess (Classical/Rapid/Blitz/Bullet with separate rating systems and tournaments) and Olympic decathlon (single-event specialists + decathlon for generalists). Most Tier A tournaments are single-format (e.g., HackLet Vibe Sprint tournament, HackLet Unslop Agile tournament). Annual or seasonal cross-format championships rotate formats across rounds. Year-end all-stars events bring top performers from each format together. Credentialing hierarchy stays clean: single-variant champion (specific format Tier A) → format overall champion (cross-variant within format) → cross-format championship (across both formats) → sustained multi-year champion. Each level signals progressively more comprehensive capability. Specialists have appropriate venues; generalists have appropriate venues; the credentialing ecosystem accommodates both pathways. *Worth landing in IDEAS as the design pattern for Tier A coordination; operational work probably 5+ years out.*

- **Chapter variant portfolio for Tier A operations (specialization is common but not required).** Tier A chapters' variant scope is determined by their verification application — a chapter applies for verification on specific {Format × Timer} variants and operates Tier A events only on those verified variants. **Specialization is a common operational pattern but not a league-imposed constraint**: chapters often concentrate Tier A verification on one variant family (e.g., Vibe Sprint + Vibe Agile across the Vibe format axis) because infrastructure investment is bounded per variant, operational expertise concentrates, community identity develops around format affinity, and the verification application work scales linearly with variant count. However, chapters with substantial operational capacity may apply for verification across multiple variants over time, and the league's data model supports any chapter portfolio. Initial Tier A chapters pick the variant that matches their community demand (BU community probably wants Vibe Sprint based on cyber/AI club's existing comfort with chat-window AI workflows); they may expand verification as infrastructure/experience grow. The cardinality is Chapter↔Format many-to-many (mediated through Events), Event↔Format 1-to-1 (each event commits to one variant per the format_spec §7.1 rule), Chapter↔Event 1-to-many. *Worth landing as Tier A operational guidance when verification application materials are drafted (probably Year 2-3).*

- **Vibe-vs-Unslop credentialing hierarchy.** The two formats credential structurally different skill clusters and map to different parts of the FDE work surface. **HackLet Vibe** (generative work — building from scratch) credentials *interview-readiness* — the "given data, design something" skill cluster that FDE interviews test for. Maps to greenfield deployment work where FDEs build new integrations in customer environments. **HackLet Unslop** (interpretive work — remediating broken code) credentials *work-readiness* — diagnosing and fixing existing problematic deployments under AI assistance. Maps to the dominant FDE work pattern: walking into customer codebases with existing AI-generated artifacts, legacy decisions, and accumulated technical debt. Per the 95% AI ROI failure rate (MIT NANDA), the FDE bottleneck in 2026 is *fixing the slop* previous AI use produced, not building more greenfield AI projects. Unslop credentials this remediation skill specifically. A candidate with Tier A wins in *both* formats has demonstrated comprehensive FDE-readiness: can pass interviews (Vibe) AND can do the work (Unslop). Hierarchy: Vibe is foundational (validates engineering judgment, AI direction, defensive coding, verification reflex through generative work); Unslop is applied (validates the same skill cluster *plus* triage, diagnosis, and remediation through interpretive work). Combined wins represent demonstrated capability across both. *Worth being explicit in eventual employer-facing materials and sponsor pitch development.*

- **Stress-equivalence credentialing-design principle (discovered benefit of 24-min compression).** A credentialing test should produce **peak stress exceeding the work it credentials**, with **failure consequences below work consequences**. HackLet's 24-minute compression — originally chosen for the 24-hour hackathon unit-substitution joke — encodes exactly this principle. Compression compensates for cumulative FDE work pressure (peak intensity vs sustained duration). Adversarial evaluation compensates for delayed external scrutiny (immediate fuzz vs delayed code review). Broadcast visibility compensates for customer-facing pressure (intense observation vs prolonged observation). Cross-examination compensates for technical defense in customer meetings (concentrated 120s vs distributed conversation). HackLet failure consequences (DNF, DQ, last place, awkward broadcast moment) recover within days; FDE failure consequences (angry customers, lost contracts, terminations, lawsuits) damage careers for years. Multi-day Tier A tournament structure adds the endurance dimension matching FDE multi-month patterns. Honest limitations: doesn't credential sustained multi-month project capability, multi-stakeholder political navigation, logistical adaptability under travel demands. The principle generalizes — every credentialing institution that works (chess, athletics, FMWC, professional certifications) follows this stress-equivalence + stakes-asymmetry pattern. *Worth being explicit in eventual sponsor and employer materials; the principle is what makes HackLet credentialing claims defensible structurally.*

- **Structural durability of HackLet's credentialing claims (substrate assumptions).** HackLet's credentialing positioning depends on LLM-based AI substrate maintaining structural properties indefinitely. The key properties: (1) Probabilistic generation produces hallucinations as inherent computational property (theoretically established via diagonalization arguments in 2025-2026 research showing any enumerable model collection must hallucinate on some inputs — *not* an engineering bug that gets fixed); (2) RLHF training paradigms induce sycophancy as a reward-hacking property persistent in frontier systems (50%+ sycophantic behavior documented across 11 leading models per Science 2026); (3) Confidence calibration remains poor — models produce confidently incorrect output and verbalize uncertainty poorly even when uncertain (overconfidence in 70-80%+ of answers documented); (4) Training cost scaling shows diminishing returns (Epoch projects deceleration from 10x/year to 3-4x/year through 2028, MMLU-Pro accuracy approaching ~95% asymptote regardless of compute); (5) Inference cost concerns persist for production deployment, making resource-bounded AI usage a permanent professional skill. While *scaffolding* evolves rapidly (tools, MCPs, RAG patterns, agentic loops, MoE routing, verifier models), the *substrate properties* remain durable. This means the skill cluster HackLet credentials (verification reflex, AI direction, defensive coding, technical communication under pressure) stays market-relevant as long as these properties hold. The credentialing "window" is structural not temporary. Unslop *gains* relevance over time as more AI-generated code accumulates in production. Format evolution discipline (quarterly substrate tier list updates, annual format evolution releases, continuous catalog development) keeps the credential current with scaffolding evolution while testing durable underlying skills. *Worth being explicit in eventual sponsor materials — transforms HackLet positioning from "current market window" to "infrastructure for the durable state of AI-era engineering work."*

- **Multi-dimensional leaderboard architecture (Tier A maturity work).** Global leaderboards display multiple metrics simultaneously rather than collapsing to a single composite ranking. For each player: cumulative score (sum across all Tier A events, normalized across catalog versions), per-event average (cumulative ÷ N), N (sample size visible), standard deviation (consistency indicator), peak single event (ceiling demonstrated), median single event (robust typical performance). Per-format breakdown (Vibe vs Unslop separate metrics), per-tier breakdown (Tier A vs Tier B vs Tier C with Tier C marked as bounded), per-catalog-version breakdown (rankings within each quarterly catalog version), aggregate cross-version rankings (normalized for cross-catalog comparability). Default sort by composite formula (cumulative weighted by sqrt(N) and consistency penalty — empirical calibration once real data exists); multi-metric detail visible on player profile view. Slop score (catalog-dependent, range [0, +∞), lower is better) normalized to a percentile against the current catalog for cross-version legibility — normalization matters more under deduction-only, since raw slop depends on catalog scope; communication score (bounded 0-100) displayable raw. The structure supports nuanced employer hiring decisions: different roles prioritize different dimensions, leaderboard supports that nuance rather than flattening to one number. *Substantial platform engineering work; realistic timing is when Tier A operations begin and aggregate rankings carry credentialing-grade weight (Year 3+). Lands in BUILD_ROADMAP Stage 8 federation features.*

- **Catalog versioning with quarterly cadence (Tier A operations).** The fuzz catalog evolves continuously to track threat landscape changes and player adaptation. Without explicit versioning, scores across catalog evolution aren't directly comparable. Catalog versions are dated and credentials carry the version: "HackLet Vibe Sprint, Tier A, Q3 2026 catalog, ranked 4th of 47." Quarterly cadence balances stability (12-week windows accumulate meaningful sample size at each version) with evolution (4 updates/year keeps catalog current). Catalog changelog documents per-version: probes added, probes deprecated, weight adjustments, scoring formula tweaks. Tournaments operating across a cutover commit to one version (the one they started on). Two parallel ranking surfaces: per-catalog-version leaderboards (Q3 2026 catalog leaderboard, Q4 2026, etc.) and aggregate cross-version leaderboards (lifetime rankings normalized across versions). Employers can answer different questions: "best at the current catalog" (per-version) or "best across catalog evolution" (aggregate). Versioning rigor scales by tier: Tier A version-strict (credentials cite catalog version), Tier B weakly versioned (catalog version available but optional in credential string), Tier C unversioned (catalog version not part of chapter-local credential by default). All tiers use the same current catalog; only credentialing scope differs by tier. *Operationally relevant when Tier A begins; canonical formalization probably Year 2-3. Matches FIDE rating cycles, FMWC scoring rules updates, sports record-keeping era conventions.*

- **Inspector Corps infrastructure (Tier A operational dependency).** Tier A events require league-appointed on-site inspectors per LEAGUE_OPERATIONS.md §4. Inspector availability is the binding operational constraint on Tier A event volume. Rough scale at maturity: 4-6 trained inspectors, each sustainably handling ~1 event/month (more risks fatigue and inspection quality), yields ~48-72 Tier A events/year league-wide. Tier A is structurally scarce — that's a feature, not a bug, because scarcity preserves credentialing weight. Inspector qualification: trained league reps who have personally operated Tier A events at their own chapter and demonstrated competence at infrastructure audit. Probably initially the league founder team; expands as the chapter ecosystem matures and produces senior chapter operators. Operational details to specify when activation approaches: training pathway and certification requirements, scheduling protocol (advance booking, cancellation policy, inspector matching to events), travel and compensation logistics, conflict-of-interest rules (inspector can't audit a chapter they're affiliated with), the appeal authority of inspector findings, the role of inspector findings in sanction decisions per §11. *Operationally relevant when Tier A verification begins (Year 3+). Documenting now so the constraint is acknowledged in roadmap planning.*

## Substrate & Catalog

- **Majority intersection substrate curation (with final Season 1 selection).** Two-clause filter: (1) language appears in ≥ N% of fresh-grad SWE *aspirational* resumes (the resumes CS undergrads write to break into entry-level web SWE — not all SWE resumes, which would dilute with COBOL maintainers); (2) league maintains fuzz coverage across all six axes for the dominant framework(s). Both required for Tier 1. The "majority intersection" framing beats union (union grows monotonically toward the universe) and beats strict intersection (approximately empty). The critical convergence property: in this domain, audience interest and testability *align* — the same forces that put a language on aspiring engineers' resumes make it testable (most training data, most security research, most framework standardization, most documentation). HackLet doesn't have to choose between "what players want" and "what we can test." Threshold for S1: N ≥ 50%.

  **Final Season 1 substrate:**

  *Tier 1 (full coverage):* Python (Flask, FastAPI, Django, Starlette), JavaScript (Express, Fastify, Hono, Koa), TypeScript (same Node ecosystem), Java (Spring Boot, Quarkus, Micronaut), Go (net/http, Gin, Echo, Chi, Fiber), PHP (Laravel, Symfony), C# (ASP.NET Core with Identity + EF Core).

  *Universal substrate:* HTML, CSS, vanilla JS, SQL (PostgreSQL, SQLite, Redis Tier 2).

  *Tier 3 (universal-only testing, diversity signaling):* C (via libmicrohttpd, mongoose), C++ (via Crow, Pistache). The universal arsenal applies because any deployed HTTP server presents interactable interfaces. Roughly 50-70% catalog coverage — meaningfully participatory, not throwaway.

  *Excluded:* Rust (below threshold for junior web roles, framework ecosystem still maturing), Ruby (declining), Swift/Kotlin (mobile-focused), niche languages.

  *Lands in format_spec.md §5.4 post-pilot. Two-clause filter generalizes to frameworks, databases, auth patterns — same principle at every substrate layer.*

- **Methodology transparency for substrate decisions.** Publish per season: data sources sampled (LinkedIn Easy Apply entry-level SWE filters, HackerRank student survey, Stack Overflow student responses, FAANG/tier-2 new grad listings, university career services data, hackathon repos), sample sizes, threshold N%, resulting Tier 1/2/3 placements. Makes curation auditable. Defends against "favoritism" criticism — "Python is N% of target resumes, well above our Y% threshold" beats "we like Python." Matches academic transparency norms.

## Broadcast & Production

- **Process-as-spectacle broadcast theory (MasterChef pattern).** The build phase IS the show, not just a precursor to pitch+crossex. AI substrate compresses the build to televisable duration. Screen capture is richer than cooking-show overhead cameras (audiences directly observe creative work). Real-time metrics provide commentary substance — without them, commentary is surface-level "she's typing fast" filler. The slop score is the commentary backbone. From MasterChef specifically: three-criteria scoring per category (substance/coherence/delivery for pitch; defense substance/code grounding/composure for cross-ex), quantitative scores within subjective evaluation, multi-judge specialization with disagreement as content, deliberation as its own broadcast phase. What we don't borrow: reality-TV theatrics, celebrity-judge personality center, elimination dynamics. *Lands in format_spec.md §3.9 when expanded; LEAGUE_OPERATIONS for the two-precedent positioning.*

- **Chat log as the killer broadcast asset.** The chat log between player and AI is HackLet's most powerful broadcast asset. Structurally novel — no previous televised creative competition has had access to the contestant's cognition stream in real time. Chess broadcasts show engine evaluation, not player thinking. Cooking shows infer chef intent. HackLet broadcasts the cognition itself. Conversational format is legible to non-technical audiences. Prompts reveal strategy without commentator explanation needed ("scratch that, just test sqli" explains itself). The player-AI dynamic creates relatable interpersonal-feeling drama. Director pattern: 12 screen captures broadcast to production; director picks the most interesting screen at each moment based on prompt activity, fuzz events, visual progress, crisis moments, late-round panic. *Picture-in-picture for parallel comparisons. Commentator dashboard surfaces story candidates. Lands in format_spec.md §3.9 when expanded.*

- **Venue vs broadcast audio separation.** Venue stays quiet (players need to concentrate). Commentary lives in a separate booth, mixed for broadcast only. Matches sports broadcast conventions (tennis, golf, chess). Operational implication for Tier A chapters: commentary booth must be sound-isolated, watching the broadcast feed not the room directly. *Lands in chapter broadcast setup docs when broadcast infrastructure is built (Stage 6).*

- **Vocabulary bootstrapping + cultural-infrastructure framing.** Each broadcast event teaches the audience more engineering spectator vocabulary. By Season 3-5, "defended SQL clean," "burning tokens fighting the model," "scratch the search endpoint" become familiar broadcast vocabulary — the way "deglaze," "fond," "ribbon stage" became familiar through MasterChef. MasterChef built cooking spectator vocabulary over 15 years; HackLet has the same opportunity for engineering, accelerated by the non-technical judge addition acting as on-broadcast vocabulary delivery. *This is the long-term cultural-infrastructure play — bigger than "another competitive format." Engineering hasn't had visible practice rituals before. HackLet positions to change that.*

The format only exists because AI exists. (2) Hackathons degrade because of AI; HackLet inverts the trajectory — hackathon credentialing weakens as AI capability grows (human contribution becomes ambiguous), HackLet credentialing strengthens (measured skills isolate to direction / verification / defense). (3) The 24-minute format makes engineering finally televisable — MasterChef proves process-as-spectacle, AI compression makes builds fit watchable durations, chat log is the cognition feed. (4) Convergence + divergence makes the format both watchable and comparable — curated substrate enforces enough convergence for apples-to-apples comparison, AI direction creates enough divergence for content.

- **The elevator pitch: "FMWC + MasterChef for vibecoding."** Two precedents handle two halves. Years 4-5 (regional/championship maturity): 10k-30k per championship. Years 6-10: ESPN8-tier broadcast deals, $100k+ prize pools, real recruiting pipeline. - **Esports-first positioning with credentialing as additive layer (foundation framing).** HackLet's primary institutional identity is **esports** — a genuinely competitive format for AI-assisted technical building, watchable as broadcast property, drawing player and audience interest through inherent appeal of the competition itself. Credentialing is the **additive** claim that exists because nothing else reliably credentials FDE-relevant skills *yet*. - **Credential reliability argument (structural anti-cheating + skill cluster + verifiability).** Tier A wins can become reliable credentials through three structural properties: (1) **Structural anti-cheating** — Tier A infrastructure (firewall isolation + RMM + ephemeral accounts + league-hosted substrate + broadcast accountability) makes faking substantially harder than alternatives. Cheating at Tier A would require defeating the infrastructure itself, which requires capability substantively similar to what the credential tests; (2) **Tight skill cluster mapping** — the format tests what FDE work actually requires (production engineering + AI integration + defensive coding + technical communication under pressure + customer-facing judgment), not adjacent skills like LeetCode/hackathons/certs; (3) **Publicly verifiable signal** — persistent rankings, broadcast archives, submission code archive create one-click employer verification vs multi-step verification of current credentials that often fails. Honest calibration: reliability is *building* not *established*. Requires 5-10 years for empirical track record. Year 1-2 strongest available credential is single Tier A win (modest signal). Year 3-5 cumulative signals (multi-time multi-format champion) become available. Year 7-10 longitudinal data (HackLet champion FDE performance) accumulates. Employer recognition takes time. Format evolution discipline required. Competition could emerge. *Worth being explicit in eventual HM-facing materials: structural properties make reliability possible, empirical track record demonstrates reliability, early HMs participate in establishment.*
